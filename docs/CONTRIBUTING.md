## Getting Started

### Prerequisites

- [Bun](https://bun.sh) >= 1.3.10
- A PostgreSQL database (local or remote)

### Setup

1. Fork and clone the repository:

   ```bash
   git clone https://github.com/your-username/modern-web-starter.git
   cd modern-web-starter
   ```

2. Install dependencies:

   ```bash
   bun install
   ```

3. Set up environment variables:

   Create the required `.env` files from the examples:
   - `apps/server/.env.dev` — Database URL, CORS origin, Better Auth secrets
   - `apps/web/.env.dev` — VITE_SERVER_URL

   > See [Environment Variables](../README.md#environment-variables) in the README for details.

4. Apply the database schema:

   ```bash
   bun run db:push
   ```

5. Start the development servers:

   ```bash
   bun run dev
   ```

   The web app runs at [http://localhost:3001](http://localhost:3001) and the API at [http://localhost:3000](http://localhost:3000).

## Development Workflow

### Branch Naming

Use descriptive branch names with a conventional prefix:

- `feature/description-of-feature`
- `fix/description-of-fix`
- `refactor/description-of-refactor`
- `docs/description-of-doc-change`

### Running During Development

```bash
bun run dev:web          # Frontend only
bun run dev:server       # Backend only
bun run check-types      # TypeScript type checking
bun run check            # Oxlint + Oxfmt (lint and format)
```

## Project Structure

This is a **Turborepo** monorepo managed with Bun:

```
modern-web-starter/
├── apps/
│   ├── web/          # React + TanStack Router frontend
│   └── server/       # Hono API backend
├── packages/
│   ├── auth/         # Better-Auth configuration
│   ├── db/           # Drizzle schema, queries, migrations
│   ├── mail/         # Postmark email service
│   ├── infra/        # Alchemy deployment config
│   ├── config/       # Shared TypeScript configs
│   └── env/          # Environment validation (T3 env)
```

### Key Locations

| What               | Where                         |
| ------------------ | ----------------------------- |
| Frontend routes    | `apps/web/src/routes/`        |
| API endpoints      | `apps/server/src/`            |
| Database schema    | `packages/db/src/schema/`     |
| Database queries   | `packages/db/src/queries/`    |
| Auth configuration | `packages/auth/src/`          |
| UI components      | `apps/web/src/components/ui/` |

## Coding Conventions

### Language & Runtime

- **TypeScript** with strict mode — no `any`, no `@ts-ignore`, no `@ts-expect-error`
- **Bun** as the runtime and package manager
- Target **Cloudflare Workers** runtime — use Workers-compatible APIs

### Workspace Imports

Use the `@modern-web-starter/*` alias for cross-package dependencies:

```typescript
import { db } from "@modern-web-starter/db";
import { auth } from "@modern-web-starter/auth";
```

### Shared Dependencies (Catalogs)

Root `package.json` defines shared dependency versions via `"catalog:"` entries.
Always use the catalog reference instead of hardcoding versions:

```json
{
  "dependencies": {
    "hono": "catalog:"
  }
}
```

### Linting & Formatting

We use **Oxlint** for linting and **Oxfmt** for formatting — **not** ESLint or Prettier.

```bash
bun run check          # Run both linter and formatter
bunx oxlint            # Lint only (oxlint is NOT in PATH, always use bunx)
bunx oxfmt --write     # Format only
```

### Frontend Conventions

- **Tailwind CSS v4** — configured via Vite plugin (no `tailwind.config.js`). Global CSS vars live in `apps/web/src/index.css`.
- **TanStack Router** — file-based routing in `apps/web/src/routes/`. **Never edit** `apps/web/src/routeTree.gen.ts` (auto-generated).
- **shadcn/ui** components live in `apps/web/src/components/ui/`.
- Use TanStack Query for async state management and data fetching.

### Database Conventions

- **Drizzle ORM** with **snake_case** naming for tables and columns.
- Schema definitions in `packages/db/src/schema/`.
- Queries in `packages/db/src/queries/`.
- Migrations in `packages/db/src/migrations/` (generated by drizzle-kit).

### API Conventions

- **Hono** as the server framework — lightweight, web-standards compliant.
- API logic is in `apps/server/` (not a separate `@modern-web-starter/api` package).
- Use Zod for request validation and OpenAPI integration.

## Commit Guidelines

We follow **Conventional Commits** for clear, structured commit messages:

```
<type>(<scope>): <description>

[optional body]
[optional footer]
```

### Types

- `feat` — A new feature
- `fix` — A bug fix
- `refactor` — Code change that neither fixes a bug nor adds a feature
- `docs` — Documentation only changes
- `style` — Formatting, missing semicolons, etc. (no production code change)
- `test` — Adding or correcting tests
- `chore` — Build process, dependencies, CI, etc.

### Scope Examples

- `web` — Frontend changes
- `server` — Backend API changes
- `db` — Database schema or queries
- `auth` — Authentication package
- `mail` — Email service
- `infra` — Deployment infrastructure
- `config` — Shared configuration
- `env` — Environment validation

### Examples

```
feat(web): add dark mode toggle to settings page
fix(server): handle null email in user creation
refactor(db): simplify subscription query
docs: update deployment guide
chore(deps): upgrade hono to v4.12
```

## Pull Request Process

1. **Create an issue** first for non-trivial changes — discuss the approach before implementing.
2. **Fork and branch** from `main`.
3. **Make your changes** following the coding conventions above.
4. **Run checks locally** before submitting:

   ```bash
   bun run check          # Lint + format
   bun run check-types    # TypeScript check
   bun run test           # Run tests
   bun run build          # Verify build
   ```

5. **Open a pull request** with a clear title and description referencing the issue.
6. **Ensure CI passes** — the workflow runs lint, build, and test on every PR.
7. **Address review feedback** — maintainers may request changes.

### PR Checklist

Before submitting, verify:

- [ ] Code follows Oxlint/Oxfmt conventions (`bun run check`)
- [ ] TypeScript types are valid (`bun run check-types`)
- [ ] Tests pass (`bun run test`)
- [ ] Build succeeds (`bun run build`)
- [ ] New code includes tests (if applicable)
- [ ] Documentation is updated (if applicable)
- [ ] No `any`, `@ts-ignore`, or `@ts-expect-error` annotations added

## Testing

```bash
bun run test            # Run all tests
bun run test:server     # Server tests only
bun run test:web        # Frontend tests only
```

We use **Vitest** as the test runner. Tests live alongside the code they test.
